전체 흐름

네 단계를 가르는 기준은 “흐름을 누가 결정하는가”다.

한 줄 비유 똑똑한 직원에게 일을 맡기는 네 가지 방식이다. ① 질문 하나에 답하게 하기(LLM) → ② 정해진 체크리스트대로 처리하게 하기(Workflow) → ③ 목표를 주고 방법·도구 선택을 맡기기(Agent) → ④ 여러 작업자와 도구의 실행 흐름을 조율하기(Orchestration).

0.1 LLM — 입력 문맥을 바탕으로 답을 생성하는 기본 단위

LLM은 입력된 문맥을 바탕으로 응답을 생성하는 모델이다. 가장 단순한 형태에서는 사람이 요청을 넣고, 모델이 한 번의 답변을 생성한다.

다만 실제 제품이나 애플리케이션에서는 LLM에 검색, 도구 호출, 메모리, 상태 관리가 붙을 수 있다. 따라서 여기서 말하는 LLM은 그런 추가 장치가 붙기 전의 기본 모델 호출 에 가깝다.

  • 비유 — 질문 하나에 답해주는 똑똑한 사람
  • 예시 — “이 문단 요약해줘” → 요약 결과를 받고, 다음 행동은 사람이 결정
  • 대표 제품 — ChatGPT · Claude · Gemini 같은 채팅형 AI
  • 대표 형태 — 채팅형 AI의 단발 응답, API 모델 호출 1회

0.2 Workflow — 사람이 설계한 고정 경로를 따르는 자동화

Workflow는 사람이 미리 설계한 경로를 따라 LLM 호출, 조건 분기, 검색, 도구 실행을 순서대로 엮는 방식이다. 무엇을 언제 실행할지는 주로 코드나 플로우 설정이 결정하고, 모델은 각 단계에서 필요한 판단이나 생성을 맡는다.

이 방식은 경로가 비교적 예측 가능하고 반복 가능하다는 장점이 있다. 고객 문의 분류, 문서 요약 파이프라인, 정해진 승인 절차처럼 처리 순서가 안정적인 업무에 잘 맞는다.

  • 비유 — 정해진 체크리스트를 따라 일하는 직원
  • 예시 — 문의 접수 → 분류 → 답변 초안 생성 → 사람 검토
  • RAG 예시 — 검색 → 문서 추출 → 요약처럼 경로가 고정된 RAG 파이프라인은 Workflow 예시로 볼 수 있다.
  • 대표 형태 — n8n · Zapier · Dify 의 고정 플로우

0.3 AI Agent — 목표를 보고 다음 행동과 도구 사용을 선택하는 시스템

AI Agent는 모델이 단순히 한 번 답하는 데서 끝나지 않고, 사용자의 목표를 이해하고 필요한 절차를 판단하며, 도구를 사용해 작업을 수행하는 AI 비서에 가깝다.

핵심 구분 경로를 코드나 사람이 미리 정하면 Workflow , 목표를 기준으로 모델이 다음 행동을 동적으로 선택하면 Agent 에 가깝다.
  • 비유 — “이 목표를 달성해와”라고 맡기면 방법과 도구를 고르는 직원
  • 예시 — “이 버그를 고쳐줘” → 파일 탐색 → 원인 추정 → 수정 → 테스트 실행 → 실패 시 재시도
  • 대표 제품 — Claude Code · Codex · Cursor 같은 coding agent
  • 대표 형태 — coding agent, research agent, tool-using assistant

0.4 Orchestration — 여러 Agent와 도구 흐름을 조율하는 구조

Orchestration은 하나의 모델 응답이나 하나의 Agent만으로 처리하기 어려운 작업을 여러 단계, 여러 도구, 여러 Agent로 나누고 전체 실행 흐름을 조율하는 구조다. 조율 주체는 코드일 수도 있고, 상위 Agent일 수도 있으며, 두 방식을 섞을 수도 있다.

따라서 Orchestration을 단순히 “여러 에이전트가 협업한다”로만 보면 조금 좁다. 더 정확히는 어떤 작업을 어떤 순서로 실행할지, 실패하면 어떻게 복구할지, 어떤 결과를 다음 단계에 넘길지 를 설계하는 영역이다.

헷갈리지 말 것 Workflow는 대체로 고정된 경로 를 실행하는 쪽에 가깝고, Orchestration은 여러 실행 단위의 순서·분기·위임·검증 을 조율하는 더 넓은 구조다.
설계 방향

작업을 하나의 모델에 몰아주지 않고 역할별로 나눌 수 있다

모델이나 도구마다 제공 환경과 강점이 다르기 때문에, 탐색·계획·구현·검증 같은 역할을 분리해 배치할 수 있다. 이때 중요한 것은 특정 모델이 항상 더 낫다는 단정이 아니라, 작업 특성에 맞게 실행 단위를 나누고 결과를 검증하는 구조다.

검증 관점

역할 분리는 품질 관리와도 연결된다

한 단계가 만든 결과를 다른 단계에서 검토하거나, 검색 결과와 생성 결과를 분리하거나, 테스트 실행을 별도 단계로 두면 오류를 더 빨리 발견할 수 있다. Orchestration은 단순히 많이 연결하는 일이 아니라, 실패 가능성을 관리하는 구조 설계에 가깝다.

  • 비유 — 매니저가 여러 전문가와 도구를 조율해 하나의 결과물을 만드는 방식
  • 예시 — 리서치 과제를 검색, 요약, 분석, 검증 단계로 나누고 결과를 종합
  • 대표 형태 — multi-agent framework, agent workflow framework, tool orchestration layer

0.5 Summary

네 단계를 하나의 축으로 보면 분명하다. 위로 갈수록 사람이 매번 직접 정하던 실행 흐름을 코드, 모델, 여러 Agent의 조율 구조가 더 많이 맡는다. 그만큼 유연성은 커지지만 예측 가능성, 비용, 검증 난이도도 함께 올라간다.

개념 토대

프롬프트는 단순한 질문이 아니라, AI가 작업을 해석하는 기준이다.

1.1 프롬프트 엔지니어링의 핵심

프롬프트 엔지니어링 은 AI에게 원하는 결과를 얻기 위해 입력 문장을 의도적으로 설계하는 일이다. 단순히 말을 예쁘게 쓰는 것이 아니라, AI가 무엇을 해야 하는지, 어떤 기준으로 판단해야 하는지, 어떤 형태로 답해야 하는지 를 정리해서 전달하는 작업이다.

같은 모델을 사용해도 프롬프트가 모호하면 결과는 흔들린다. 반대로 역할, 목적, 맥락, 제약, 출력 형식이 명확하면 모델을 바꾸지 않고도 결과 품질을 크게 끌어올릴 수 있다.

핵심 프롬프트는 AI에게 던지는 질문이 아니라, AI가 작업을 수행할 때 따라야 하는 작업 지시서 에 가깝다.

1.2 LLM이 답을 만드는 방식

LLM은 데이터베이스처럼 정답을 꺼내오는 시스템이 아니다. 입력된 문맥을 바탕으로 다음에 올 가능성이 높은 토큰을 예측 하면서 답변을 생성한다.

그래서 AI는 사용자의 의도를 자동으로 완벽하게 알아차리지 못한다. 우리가 생략한 조건은 모델이 추측하고, 모호한 표현은 모델이 임의로 해석한다.

예시 정리해줘 라고 하면 요약인지, 표 정리인지, 발표용 문장인지 AI가 추측한다.
반면 회의 내용을 5줄 요약 + 결정사항 + 담당자별 할 일 표로 정리해줘 라고 하면 결과의 방향이 훨씬 명확해진다.

즉 프롬프트 엔지니어링은 모델을 속이는 기술이 아니라, 모델이 추측해야 하는 영역을 줄이는 기술이다.

1.3 좋은 프롬프트의 구성요소

좋은 프롬프트는 길다고 좋은 것이 아니다. 필요한 요소가 빠지지 않고, 서로 충돌하지 않으며, 결과물을 판단할 기준이 있어야 한다.

  • 목적 — 왜 이 작업을 하는지, 결과물을 어디에 쓸지 알려준다.
  • 역할 — 어떤 관점으로 답해야 하는지 정한다. 예: 강사, 리뷰어, 기획자, 개발자.
  • 맥락 — 대상, 배경, 현재 상황, 참고해야 할 정보를 제공한다.
  • 작업 지시 — 무엇을 해야 하는지 구체적으로 적는다. 예: 요약, 비교, 수정, 코드 작성.
  • 제약 조건 — 제외할 것, 지켜야 할 것, 주의할 범위를 정한다.
  • 출력 형식 — 표, 목록, 문단, JSON, 코드 등 원하는 결과 모양을 지정한다.
템플릿 역할 : 너는 ○○ 전문가다.
맥락 : 현재 상황은 ○○이다.
작업 : ○○을 해줘.
제약 : ○○은 하지 말고, ○○ 기준을 지켜줘.
출력 : 결과는 ○○ 형식으로 작성해줘.
02

핵심 기법

프롬프트를 잘 쓴다는 건 문장을 길게 쓰는 게 아니라, 모델이 덜 추측하게 만드는 것이다.

2.1 명확하고 구체적으로 지시하기

가장 기본이지만 가장 자주 무너지는 부분이다. 모델은 우리가 머릿속으로 기대한 배경을 자동으로 알지 못한다. 그래서 대상, 범위, 목적, 기준 을 명시하지 않으면 모델은 빈칸을 스스로 추측해서 채운다.

나쁜 예 이 내용 정리해줘
좋은 예 회의록을 팀장 보고용 으로 정리해줘.
핵심 결정사항, 담당자별 액션아이템, 미해결 이슈를 나눠서 작성하고,
불확실한 내용은 추측하지 말고 확인 필요 로 표시해줘.

좋은 프롬프트는 길어서 좋은 게 아니라, 모델이 어떤 기준으로 답해야 하는지 를 분명하게 알려줘서 좋다.

2.2 역할·페르소나 부여

역할을 주는 이유는 단순히 분위기를 바꾸기 위해서가 아니다. 역할은 모델의 관점, 어휘, 판단 기준 을 좁혀주는 장치다.

예를 들어 같은 문서를 검토하더라도, 개발자 관점과 기획자 관점, 보안 담당자 관점은 보는 기준이 다르다. 역할을 명시하면 모델은 어떤 기준으로 읽고 판단해야 하는지 더 빨리 잡는다.

  • 기획자 역할 — 사용자 흐름, 요구사항 누락, 정책 불명확성을 본다.
  • 개발자 역할 — 구현 난이도, 데이터 구조, 예외 처리, API 흐름을 본다.
  • 리뷰어 역할 — 논리 충돌, 표현 모호성, 빠진 전제 조건을 본다.
주의 역할만 주고 작업 기준을 안 주면 효과가 약하다. 너는 전문가야 보다 보안 담당자 관점에서 권한·개인정보·로그 노출 위험을 검토해줘 가 훨씬 낫다.

2.3 예시로 가르치기 (Zero-shot / Few-shot)

모델에게 설명만 주는 것을 Zero-shot , 원하는 입력·출력 예시를 몇 개 보여주는 것을 Few-shot 이라고 볼 수 있다.

Few-shot이 유용한 이유는 말로 설명하기 어려운 톤, 포맷, 판단 기준 을 예시 하나로 바로 전달할 수 있기 때문이다. 특히 문장 스타일, 분류 기준, 표 형태, 요약 방식처럼 정답보다 형식 일관성이 중요한 작업에 효과적이다.

예시 아래 형식으로 정리해줘.

입력: “버튼 클릭 시 화면이 멈춤”
출력: [UI 오류] 버튼 클릭 후 응답 없음 / 재현 조건 확인 필요

입력: “로그인이 가끔 안됨”
출력: [인증 오류] 간헐적 로그인 실패 / 발생 시간·계정 조건 확인 필요

예시는 모델에게 “이런 식으로 답하라”는 가장 강력한 힌트다. 설명보다 예시가 더 빠르게 먹히는 경우가 많다.

2.4 Chain of Thought(CoT) — 단계적 추론 유도

Chain of Thought(CoT) 는 모델에게 바로 결론을 요구하지 않고, 문제를 해결하기 위한 판단 기준과 중간 추론 단계를 구조화하도록 유도하는 프롬프트 기법이다.

단순히 “1번, 2번, 3번 순서로 처리해줘”라고 작업 순서를 나누는 것과는 다르다. CoT의 목적은 처리 절차를 늘리는 것이 아니라, 어떤 기준으로 판단했고, 어떤 근거로 결론에 도달했는지 를 검토 가능한 형태로 만들도록 유도하는 데 있다.

구분 중심 질문 예시
단계적 처리 유도 무엇을 먼저 하고, 다음에 무엇을 할 것인가 요약 → 분류 → 표 작성
Chain of Thought(CoT) 어떤 기준과 근거로 결론에 도달할 것인가 판단 기준 설정 → 기준별 검토 → 근거 요약 → 최종 결론

예를 들어 요구사항 검토라면 “요약해줘”보다 아래처럼 판단 기준과 근거 구조를 함께 지정하는 편이 낫다.

프롬프트 패턴 요구사항을 바로 결론 내리지 말고 다음 순서로 검토해줘.
1. 먼저 판단 기준을 정리한다.
2. 각 기준에 따라 요구사항을 검토한다.
3. 기준별 근거를 짧게 요약한다.
4. 마지막으로 최종 결론과 추가 확인이 필요한 항목을 분리해 제시한다.

이 방식은 복잡한 분석·비교·검토 작업에서 특히 유용하다. 모델의 답변이 단순 결론으로 끝나지 않고, 판단 기준, 검토 흐름, 최종 근거 를 함께 포함하기 때문이다.

주의 CoT를 “내부 생각 과정을 전부 보여달라”는 의미로 쓰면 안 된다. 실무에서는 긴 사고 과정 전체보다 검토 가능한 단계, 판단 기준, 요약된 근거, 최종 결론 이 드러나는 출력 형식이 더 유용하다.

2.5 출력 형식 지정

좋은 답변은 내용뿐 아니라 사용하기 쉬운 형태 여야 한다. 같은 내용이라도 보고용 문장, 표, 체크리스트, JSON, 마크다운에 따라 활용도가 달라진다.

출력 형식을 지정하지 않으면 모델은 그때그때 적당한 형식을 선택한다. 하지만 실무에서는 복사해서 붙여넣거나, 문서에 반영하거나, 코드에 연결해야 하므로 출력 형식을 명확히 정하는 것이 중요하다.

  • 보고용 — 배경, 핵심 내용, 이슈, 다음 액션 순서로 작성한다.
  • 검토용 — 항목, 문제점, 영향도, 수정 제안으로 나눈다.
  • 개발용 — 입력값, 처리 조건, 예외, 출력값을 구조화한다.
  • 자동화용 — JSON, YAML, CSV처럼 파싱 가능한 형식을 지정한다.
팁 지시문과 참고 데이터를 구분자나 태그로 나누면 좋다. 예: <요청> , <자료> , <출력형식> . 이렇게 하면 모델이 어디까지가 명령이고 어디부터가 참고자료인지 덜 헷갈린다.

2.6 제약과 금지사항 지정

프롬프트는 “무엇을 해라”만 적는 것이 아니다. “무엇은 하지 말라”도 함께 적어야 한다. 모델은 친절하게 답하려는 경향이 있어서, 정보가 부족해도 그럴듯하게 채우거나 요청 범위를 넓혀버릴 수 있다.

그래서 실무 프롬프트에는 아래 같은 제약이 자주 필요하다.

  • 범위 제한 — 제공한 자료 안에서만 답하라.
  • 추측 금지 — 근거가 없으면 “확인 필요”로 표시하라.
  • 형식 제한 — 표로만 작성하라, 5줄 이내로 작성하라.
  • 대상 제한 — 비개발자도 이해할 수 있게 설명하라.
핵심 원하는 결과를 얻으려면 해야 할 일 과 하지 말아야 할 일 을 같이 줘야 한다. 제약은 모델을 답답하게 만드는 것이 아니라, 결과를 사용할 수 있게 만드는 안전장치다.
03

개선 루프와 신뢰성 관리

프롬프트는 한 번에 완성하는 문장이 아니라, 결과를 보고 조정하는 작업 절차다.

3.1 첫 출력은 완성본이 아니라 초안이다

프롬프트 엔지니어링에서 중요한 태도는 첫 답변을 바로 최종 결과로 보지 않는 것이다. 같은 모델이라도 지시의 구체성, 예시의 품질, 출력 형식, 제공된 맥락에 따라 결과가 달라질 수 있으므로 첫 출력은 프롬프트가 충분했는지 확인하는 피드백 으로 보는 편이 안전하다.

예를 들어 “회의록 정리해줘”라는 요청으로 나온 결과가 너무 장황하다면, 단순히 모델이 별로라고 판단하기보다 어떤 조건이 빠졌는지 확인해야 한다. 대상 독자, 요약 길이, 의사결정 사항, 액션 아이템, 제외할 잡담 여부가 빠졌다면 다음 프롬프트에 그 조건을 추가한다.

개선 루프 요청한다 → 결과를 본다 → 빠진 조건을 찾는다 → 프롬프트를 수정한다 → 다시 비교한다.
이 반복을 통해 프롬프트는 단순한 질문에서 재사용 가능한 작업 지시서로 바뀐다.

3.2 실패 원인을 프롬프트 관점에서 분해한다

결과가 마음에 들지 않을 때 “AI가 못했다”로 끝내면 개선이 어렵다. 프롬프트 관점에서는 실패를 몇 가지 원인으로 나눠 보는 것이 좋다. 원인을 나누면 다음 프롬프트에서 무엇을 고쳐야 하는지가 명확해진다.

실패 유형 증상 수정 방향
목표 불명확 요약인지 분석인지, 초안 작성인지 검토인지 결과가 흔들린다. “무엇을 해야 하는지”를 동사 중심으로 명확히 쓴다. 예: 요약해줘, 비교해줘, 오류를 찾아줘.
맥락 부족 대상 독자, 업무 배경, 판단 기준이 반영되지 않는다. 사용 목적, 독자 수준, 프로젝트 상황, 참고 자료를 추가한다.
출력 형식 부재 문단, 표, 목록이 섞이고 복사해서 쓰기 어렵다. 표, bullet, JSON, 보고서 문체처럼 원하는 구조를 지정한다.
제약 조건 부재 원치 않는 설명이 붙거나 범위 밖 내용을 추측한다. 하지 말 것, 제외할 내용, 모르면 모른다고 할 조건을 함께 적는다.

이렇게 실패 유형을 나누면 프롬프트 수정이 감으로 하는 일이 아니라, 조건을 보강하는 작업이 된다.

3.3 환각은 제거가 아니라 관리 대상이다

LLM은 주어진 입력을 바탕으로 그럴듯한 답변을 생성하는 모델이기 때문에, 사실 확인이 필요한 작업에서는 존재하지 않는 정보나 부정확한 내용을 그럴듯하게 말할 수 있다. 따라서 프롬프트로 환각을 완전히 없앤다고 표현하면 과장이다. 더 정확하게는 환각 가능성을 낮추고, 검증 가능한 형태로 답하게 만드는 것 이다.

실무에서는 답변을 생성하게 하는 것보다 답변의 근거를 함께 남기게 하는 방식이 더 안전하다. 특히 최신 정보, 법률·의료·재무처럼 정확도가 중요한 주제, 내부 문서 기반 답변에서는 모델이 알고 있는 것처럼 말하지 않도록 범위를 좁혀야 한다.

  • 근거 범위 제한 — “아래 제공한 문서 안에서만 답해줘”처럼 답변 근거를 제한한다.
  • 모르면 모른다고 하기 — 문서에 없으면 “문서에서 확인 불가”라고 답하게 한다.
  • 출처와 판단 분리 — 사실, 추론, 가정을 구분해서 쓰게 한다.
  • 검증 가능한 출력 — 중요한 주장에는 근거 문장, 출처, 항목 번호를 함께 요구한다.
주의 “절대 거짓말하지 마” 같은 지시만으로는 충분하지 않다. 근거 자료를 제공하고, 답변 범위를 제한하고, 확인 불가 시의 대체 응답을 지정하는 편이 더 안전하다.

01

Context Engineering 개념 토대

1.1 Context Engineering의 핵심

많은 사람이 AI를 사용할 때 먼저 프롬프트 를 떠올린다. 그래서 같은 요청을 더 자세히, 더 길게, 더 정교하게 쓰려고 한다.

예시 로그인 기능 만들어줘
→ React로 로그인 기능 만들어줘
→ TypeScript로 JWT 기반 로그인 기능 만들어줘

하지만 실제 프로젝트에서는 프롬프트 한 줄보다 더 중요한 것이 있다. AI가 프로젝트에 대해 무엇을 알고 있는가 다.

같은 “로그인 기능 만들어줘”라는 요청이라도, 프로젝트 구조·기술 스택·코딩 컨벤션·인증 방식·API 규칙·보안 정책을 알고 있는 AI와, 아무것도 모르는 AI는 전혀 다른 결과를 만든다.

Prompt Engineering 이 “이 요청을 어떻게 표현할까”를 고민하는 일이라면, Context Engineering 은 “AI가 성공하려면 무엇을 알아야 하고, 무엇을 덜어내야 할까”를 설계하는 일이다.

비유 세계적인 셰프(=강력한 LLM)에게 “생일파티 음식 해줘”라고만 하면 맛은 있겠지만 손님 기대와 다를 수 있다. 인원수·알레르기·선호·행사 성격을 알려주는 것 — 그게 Context Engineering이다.

1.2 Context rot — 많이 넣을수록 좋은 게 아니다

처음에는 이렇게 생각하기 쉽다. “그럼 프로젝트 문서를 전부 넣으면 되는 거 아닌가?”

하지만 컨텍스트는 많을수록 항상 좋아지는 것이 아니다. 오래된 정보, 서로 충돌하는 정보, 이미 의미가 없어진 정보가 섞이면 오히려 AI의 판단 품질이 떨어질 수 있다.

이처럼 컨텍스트가 길어지고 복잡해질수록 모델의 집중력과 정확도가 떨어지는 현상을 Context rot 이라고 볼 수 있다.

1.3 컨텍스트 자산 관리

좋은 컨텍스트는 즉흥적으로 매번 새로 쓰는 것이 아니라, 프로젝트 안에서 자산처럼 관리 되어야 한다.

  • AGENTS.md (규칙 파일) — AI에게 프로젝트를 처음 소개하는 온보딩 문서. 언어·스타일·금지사항·디렉터리 구조·작업 규칙을 담아둔다.
  • 스펙 문서 — 무엇을 만들지, 어떤 조건을 만족해야 하는지 정리한 문서. AI에게 기능의 의도를 전달한다.
  • 테스트 코드 — 무엇이 성공인지 검증하는 실행 가능한 기준. AI가 만든 결과가 맞는지 판단하게 해준다.
  • 토큰 예산 — 컨텍스트 창을 무작정 채우지 않고, 필요한 정보만 남기기 위한 관리 기준. 오래된 내용은 요약·정리·삭제한다.
핵심 Context Engineering은 문서를 많이 만드는 일이 아니다. AI가 작업할 때 필요한 정보가 적절한 위치에, 적절한 형태로, 적절한 양만큼 존재하도록 관리하는 일이다.

1.4 연결 고리

Context Engineering은 완전히 새로운 개발 방법론이라기보다, 기존 개발 방법론을 AI가 이해할 수 있는 컨텍스트로 재구성하는 관점 에 가깝다.

대표적인 예가 SDD 와 TDD 다.

SDD는 무엇을 만들어야 하는가 를 스펙으로 고정한다.
TDD는 무엇이 성공인가 를 테스트로 고정한다.

결국 둘 다 사람의 머릿속에 있던 정보를 AI가 읽고 활용할 수 있는 컨텍스트로 바꾸는 방식이다.

Context Engineering의 대표적인 실천 방식으로 SDD와 TDD를 살펴본다.

02

SDD — Spec-Driven Development

의도를 문서로 고정하고, 코드는 그 문서에서 만들어낸다.

2.1 SDD란 무엇인가

SDD(Spec-Driven Development) 는 코드를 먼저 작성하는 것이 아니라, 먼저 스펙(Spec) 을 작성하고 그 스펙을 기준으로 개발을 진행하는 방식이다.

여기서 스펙은 단순한 참고 문서가 아니다. AI 개발 환경에서는 스펙이 곧 에이전트에게 전달되는 핵심 컨텍스트가 된다.

핵심 SDD에서는 스펙이 원천 산출물 이고, 코드는 그 스펙을 바탕으로 만들어지는 결과물 에 가깝다.

즉 “코드를 고친다”보다 “스펙을 고치고, 그 스펙에 맞게 코드를 다시 정렬한다”는 사고방식에 가깝다.

2.2 왜 SDD가 필요한가

AI에게 “회원가입 기능 만들어줘”라고만 하면 AI는 알아서 코드를 만들 수 있다. 하지만 실제 프로젝트에서는 이 방식이 쉽게 흔들린다.

  • 요구사항이 불명확함 — 이메일 중복 체크가 필요한지, 비밀번호 정책은 어떤지, 실패 메시지는 어떻게 보여줄지 빠질 수 있다.
  • 문서와 코드가 어긋남 — 처음에는 문서대로 만들었지만, 수정이 반복되면서 코드만 바뀌고 문서는 낡는다.
  • AI가 매번 다르게 판단함 — 명확한 기준이 없으면 같은 요청에도 다른 구조와 다른 구현이 나온다.
  • 작업 범위가 쉽게 커짐 — 작은 기능을 요청했는데 AI가 화면, API, DB, 예외처리까지 임의로 확장할 수 있다.

SDD는 이런 문제를 줄이기 위해 “무엇을 만들지”를 먼저 고정한다. 그리고 그 고정된 스펙을 AI가 계속 참조하게 만든다.

2.3 SDD의 일반 산출물

SDD의 산출물은 도구마다 이름이 다르다. 따라서 constitution.md , spec.md , plan.md , tasks.md 같은 파일명을 SDD의 표준으로 보면 안 된다.

일반화하면 SDD의 핵심 산출물은 아래 4단계로 볼 수 있다.

Requirements
무엇을 요구사항 — 사용자가 원하는 기능, 해결해야 할 문제, 수용 기준을 정리한다.
↓
Design
어떻게 설계 — 화면 흐름, 데이터 구조, API, 주요 처리 방식을 정리한다.
↓
Tasks
어떤 순서로 작업 분해 — 구현 가능한 작은 작업 단위로 나눈다.
↓
Code · Test
결과물 위 산출물을 기준으로 만들어지는 코드와 테스트.
정리 SDD의 본질은 특정 파일명이 아니라 요구사항 → 설계 → 작업 → 구현 의 흐름을 문서화하고, 그 문서를 AI가 읽을 수 있는 컨텍스트로 유지하는 것이다.

2.4 도구별 구현 예시

SDD는 하나의 고정된 파일 구조를 의미하지 않는다. 도구나 팀의 방식에 따라 산출물 이름과 개수는 달라질 수 있다.

구분 산출물 예시 설명
일반화 requirements → design → tasks → code SDD를 가장 단순하게 표현한 흐름. 요구사항을 설계와 작업으로 구체화한 뒤 구현한다.
Spec Kit 계열 constitution.md → spec.md → plan.md → tasks.md 원칙 문서, 기능 스펙, 구현 계획, 작업 목록을 분리해서 관리하는 구조.
Kiro 계열 requirements.md → design.md → tasks.md 요구사항, 설계, 작업 목록 중심으로 단순화한 구조.
팀 자체 방식 PRD.md → architecture.md → todo.md 파일명은 달라도 요구사항과 설계, 작업 기준을 AI가 읽을 수 있게 관리하면 SDD 방식으로 볼 수 있다.

2.5 SDD Workflow

SDD의 흐름은 단순하다. 먼저 의도를 문서로 고정하고, 그 문서를 기준으로 설계와 작업을 나눈 뒤, 마지막에 코드를 만든다.

  • 1단계: 요구사항 작성 — 어떤 문제를 해결할지, 사용자는 무엇을 기대하는지, 완료 기준은 무엇인지 정리한다.
  • 2단계: 설계 작성 — 요구사항을 어떻게 구현할지 구조를 잡는다. 데이터, API, 화면 흐름, 예외 처리를 정의한다.
  • 3단계: 작업 분해 — AI나 개발자가 실행할 수 있는 작은 단위로 쪼갠다.
  • 4단계: 구현 — 작업 목록을 기준으로 코드를 작성한다.
  • 5단계: 피드백 반영 — 코드만 고치는 것이 아니라, 필요하면 스펙부터 수정한다.
중요 SDD에서 중요한 습관은 코드 수정 전에 스펙을 먼저 확인하는 것 이다. 스펙이 틀렸으면 코드를 직접 고치는 것이 아니라 스펙을 고치고, 그 기준에 맞게 코드를 다시 정렬한다.

2.6 장점과 한계

장점

AI가 작업 기준을 잃지 않는다

요구사항과 설계가 문서로 고정되어 있으므로, AI가 임의로 기능을 확장하거나 다른 방향으로 구현할 가능성이 줄어든다.


변경 이력이 명확해진다

기능이 바뀌었을 때 코드만 바뀌는 것이 아니라 스펙도 함께 바뀐다. 그래서 나중에 왜 그렇게 만들었는지 추적하기 쉽다.

한계

초기 작성 비용이 있다

작은 기능에도 요구사항과 설계를 먼저 정리해야 하므로, 무작정 코드부터 만드는 방식보다 처음에는 느리게 느껴질 수 있다.


시각 설계는 별도 관리가 필요하다

SDD는 텍스트 기반 스펙에 강하지만, 와이어프레임이나 화면 디자인 자체를 완전히 대체하지는 못한다. UI는 Figma, 화면설계서, 이미지 첨부 등과 연결해서 다루는 편이 현실적이다.

03

TDD — Test-Driven Development

SDD가 문서로 의도를 고정한다면, TDD는 실행되는 테스트로 검증을 고정한다.

3.1 개념: Red-Green-Refactor

테스트를 먼저 쓰고 코드를 나중에 쓴다. 테스트는 자연어 명세처럼 작동해 AI에게 측정 가능한 타겟을 준다 — “로그인 버튼 만들어”는 도박이지만, 비활성·로딩·오류 상태까지 정의한 테스트는 구체적 목표다.

Red

아직 실패하는 테스트를 먼저 쓴다. AI에게 줄 목표를 박는다.

Green

그 테스트를 통과하는 최소 코드를 작성한다.

Refactor

테스트를 green으로 유지하며 코드를 정리한다.

3.2 TDD의 산출물

SDD의 산출물이 문서라면, TDD의 대표 산출물은 실행되는 명세(테스트) 그 자체 다.

  • 테스트 목록 — 검증할 동작들의 목록. 계획용 산출물(SDD의 수용 기준에 대응).
  • 테스트 스위트 (핵심) — 단위 테스트(개발자 TDD)와 인수 테스트(ATDD/BDD). 실행 가능한 명세이자 본체.
  • 부속물 — 테스트 더블(목·스텁), 픽스처·테스트 데이터, 커버리지 리포트.
  • 구현 코드 — 테스트를 통과시키며 만들어지는 빌드 출력.

3.3 테스트 = 실행가능한 명세

단위 테스트는 프로젝트에 남는 영속 산출물이고, 머지·수정할 때마다 통과해야 하므로 항상 코드와 일치하는 “살아있는 문서” 가 된다. 요구가 바뀌면 테스트도 바뀌어 늘 최신이다.

대부분의 개발자는 작성된 문서보다 코드로 작업하길 선호하는데, 잘 쓰인 테스트가 그 함수의 동작하는 명세를 제공해 기술 문서의 상당 부분을 대신한다.

3.4 AI 개발에서 TDD가 더 중요해지는 이유

AI에게 코드를 맡기면 구현 속도는 빨라진다. 하지만 속도가 빨라질수록 더 중요해지는 질문이 있다.

질문 이 코드가 정말 요구사항에 맞게 동작하는가?

사람이 직접 코드를 작성할 때는 구현 의도가 어느 정도 머릿속에 남아 있다. 하지만 AI가 코드를 생성하면, 결과물은 빠르게 나오지만 그 코드가 어떤 기준을 만족해야 하는지는 흐려질 수 있다.

이때 테스트는 AI에게 명확한 성공 기준을 제공한다. 자연어 설명만으로는 놓치기 쉬운 조건을 실행 가능한 형태로 고정해준다.

핵심 AI 개발에서 TDD는 단순히 버그를 찾는 도구가 아니다.
AI가 만든 코드가 요구사항에 맞는지 판단하는 기준 이자, 다음 수정에서도 기능이 깨지지 않게 막아주는 안전장치 다.

예를 들어 “이메일 유효성 검사를 구현해줘”라고 요청하면 AI는 그럴듯한 코드를 만들 수 있다. 하지만 어떤 이메일을 허용하고, 어떤 입력을 거부해야 하는지는 테스트가 있어야 명확해진다.

  • 정상 입력 — 올바른 이메일 형식이면 통과해야 한다.
  • 잘못된 입력 — @ 가 없거나 도메인이 없으면 실패해야 한다.
  • 빈 값 — 아무 값도 입력하지 않으면 실패해야 한다.
  • 회귀 방지 — 이후 코드를 수정해도 기존 검증 규칙은 깨지면 안 된다.

즉 AI가 코드를 더 많이 만들수록, 테스트는 “개발자가 확인하는 도구”를 넘어 “AI에게 기준을 알려주는 컨텍스트”가 된다.

3.5 SDD와 TDD의 연결

SDD와 TDD는 따로 떨어진 방식이 아니다. 오히려 함께 쓸 때 효과가 커진다.

연결 구조 SDD는 무엇을 만들지 를 정리한다.
TDD는 그게 제대로 만들어졌는지 를 검증한다.

즉 SDD의 수용 기준은 TDD의 테스트 케이스로 이어질 수 있다.

예를 들어 SDD의 스펙에 “이메일 형식이 올바르지 않으면 가입할 수 없다”는 수용 기준이 있다면, TDD에서는 이것이 바로 실패 테스트가 된다.

그래서 AI 개발에서는 SDD로 의도를 고정하고, TDD로 결과를 검증하는 조합이 자연스럽다.

3.1

MCP 이해하기

MCP는 Model Context Protocol의 약자다. AI 애플리케이션과 외부 시스템 사이에 표준화된 연결 통로를 만들어, 모델이 파일·DB·API·브라우저·업무 도구를 사용할 수 있게 한다.

1 개념

MCP는 AI 애플리케이션이 외부 시스템에 연결되는 방식을 표준화한 오픈 프로토콜이다. 기존에는 AI가 GitHub, 데이터베이스, 파일 시스템, 검색 엔진 같은 외부 자원에 접근하려면 각 서비스마다 별도 연동을 만들어야 했다. MCP는 이 연결 방식을 하나의 규격으로 맞춰, MCP를 지원하는 클라이언트가 MCP 서버를 통해 도구와 데이터를 사용할 수 있게 한다.

핵심 MCP를 한 줄로 정리하면 “AI Agent가 외부 도구와 데이터에 접근하기 위한 공통 연결 규격” 이다.

MCP 공식 문서는 MCP를 AI 애플리케이션을 로컬 파일, 데이터베이스, 검색 엔진, 계산기, 워크플로 같은 외부 시스템에 연결하는 오픈소스 표준으로 설명한다.

2 용도

MCP는 사용자가 매번 정보를 복사해서 AI에게 붙여넣는 상황을 줄이기 위해 사용한다. 예를 들어 Claude Code가 이슈 트래커, 모니터링 대시보드, 데이터베이스, Figma, Slack, Gmail 같은 시스템과 연결되면, 모델은 사용자가 붙여넣은 텍스트만 보는 것이 아니라 실제 연결된 시스템의 정보를 읽고 필요한 작업을 수행할 수 있다.

  • 개발 업무 — GitHub 이슈, PR, 코드 저장소, 테스트 로그를 읽고 작업 맥락을 파악한다.
  • 문서·지식 검색 — 공식 문서, 사내 위키, Notion, Google Drive 같은 자료를 검색해 답변 근거로 사용한다.
  • 데이터 조회 — PostgreSQL, SQLite, BigQuery 같은 데이터 소스에서 필요한 값을 조회한다.
  • 브라우저·디자인 연동 — 웹페이지, Figma, 브라우저 자동화 도구를 통해 UI나 화면 맥락을 확인한다.
  • 업무 자동화 — 메일 초안 작성, 캘린더 확인, 티켓 생성, 알림 시스템 연동 같은 반복 작업을 연결한다.
주의 MCP는 편리하지만 권한이 커질 수 있다. 파일, DB, API, 브라우저에 접근하는 서버를 연결할 때는 읽기 전용 모드, 최소 권한, 인증 방식, 로그 노출 여부를 반드시 확인해야 한다.

3 종류

MCP 서버는 연결 대상에 따라 여러 유형으로 나눌 수 있다. 중요한 것은 “어떤 서버가 가장 유명한가”보다 “내 Agent가 어떤 외부 자원에 접근해야 하는가”다.

Claude Code 예시 01

GitHub MCP 서버 연결

GitHub 이슈, PR, repository 내용을 Claude Code에서 직접 조회하고 싶을 때 사용하는 예시다. GitHub 공식 MCP 서버는 Docker 실행 방식을 제공하며, Personal Access Token 권한은 필요한 범위만 부여하는 것이 좋다. 

# 1) GitHub Personal Access Token을 환경변수로 준비
        export GITHUB_PAT="ghp_xxxxxxxxxxxxxxxxxxxx"

        # 2) Claude Code에 GitHub MCP 서버 추가
        claude mcp add github   -e GITHUB_PERSONAL_ACCESS_TOKEN=$GITHUB_PAT   -- docker run -i --rm   -e GITHUB_PERSONAL_ACCESS_TOKEN   ghcr.io/github/github-mcp-server

        # 3) 연결 확인
        claude mcp list

        # 4) Claude Code 안에서 상태 확인
        /mcp
활용 예: “이 repository의 open issue를 보고 우선순위를 정리해줘”, “최근 PR의 변경 파일과 리뷰 포인트를 요약해줘”.
Claude Code 예시 02

로컬 stdio MCP 서버 연결 패턴

파일시스템, 사내 스크립트, 자체 Python 서버처럼 로컬 프로세스로 실행되는 MCP 서버를 붙일 때 쓰는 기본 패턴이다. 

# 기본 구조
        claude mcp add --transport stdio my-server -- npx -y my-mcp-server

        # 환경변수가 필요한 서버 예시
        claude mcp add --transport stdio internal-api   --env INTERNAL_API_KEY=your-key   -- python server.py --port 8080

        # 등록된 서버 확인 / 제거
        claude mcp get internal-api
        claude mcp remove internal-api
로컬 stdio 서버는 사용자의 PC에서 직접 실행되므로 파일 접근 범위, 실행 명령, 환경변수 노출 여부를 먼저 확인해야 한다.
Claude Code 예시 03

DB 조회용 MCP 연결 패턴

운영 DB를 직접 쓰기 권한으로 연결하기보다, 분석용 read-only 계정과 제한된 schema로 연결하는 구성이 안전하다. 

# PostgreSQL 분석용 read-only 연결 예시
        claude mcp add --transport stdio db   -- npx -y @bytebase/dbhub   --dsn "postgresql://readonly:pass@prod.db.com:5432/analytics"

        # Claude에게 요청할 수 있는 예시
        # - orders 테이블 schema를 보여줘
        # - 이번 달 매출 합계를 확인해줘
        # - 최근 90일간 구매가 없는 고객 조건을 쿼리로 정리해줘

        command = "npx"
        args = ["-y", "@upstash/context7-mcp"]

        args = [
          "run",
          "-i",
          "--rm",
          "-e",
          "GITHUB_PERSONAL_ACCESS_TOKEN",
          "ghcr.io/github/github-mcp-server"
        ]

        [mcp_servers.github.env]
        GITHUB_PERSONAL_ACCESS_TOKEN = "${GITHUB_PAT}"
프로젝트별 설정은 trusted project에서만 쓰고, token 같은 민감정보는 가능한 한 환경변수로 분리한다.
Codex CLI 예시 03

        # 더 좁게 개별 tool만 허용하는 패턴 예시
        docker run -i --rm   -e GITHUB_PERSONAL_ACCESS_TOKEN=$GITHUB_PAT   -e GITHUB_TOOLS="get_file_contents,issue_read"   ghcr.io/github/github-mcp-server

3.2

Skills 이해하기

Skills는 Agent가 반복 업무를 같은 기준과 절차로 수행하도록 만드는 재사용 가능한 작업 패키지다.

핵심 관점 MCP가 외부 도구와 데이터를 연결하는 방식이라면, Skills는 프로젝트별 작업 기준·절차·참고자료·스크립트를 묶어 재사용하는 방식 이다. 따라서 남이 만든 Skill은 그대로 쓰기보다 구조와 작성 방식을 참고하고, 실제 적용은 프로젝트 규칙에 맞춰 직접 조정하는 것이 좋다.
1

개념

Skills는 매번 길게 설명하던 작업 방식을 하나의 재사용 가능한 단위로 분리하는 방법이다.

Skill은 특정 작업을 잘 수행하기 위해 필요한 지침, 참고자료, 템플릿, 선택적 스크립트 를 하나의 폴더로 묶은 구조다. Agent는 작업 요청이 Skill의 설명과 맞는다고 판단하면 해당 Skill을 읽고, 그 안에 정의된 기준과 절차를 따라 작업한다.

핵심은 “프롬프트를 길게 잘 쓰는 것”이 아니라, 반복되는 업무를 명시적인 작업 방식 으로 만들어 두는 것이다. 이렇게 하면 코드 리뷰, 테스트 생성, 요구사항 분석, 배포 점검처럼 매번 같은 기준이 필요한 업무를 더 일관되게 처리할 수 있다.

핵심 좋은 Skill은 작업 설명만 담지 않는다. 언제 사용할지, 어떤 입력을 받을지, 어떤 순서로 판단할지, 어떤 결과물을 만들지 까지 함께 정의한다.
Basic Structure

Skill 기본 폴더 구조

공통적으로 SKILL.md 가 중심이고, 필요한 경우 스크립트·참고문서·템플릿을 함께 둔다. 

my-skill/
        ├── SKILL.md          # 필수: 메타데이터 + 작업 지침
        ├── scripts/          # 선택: 반복 실행할 스크립트
        ├── references/       # 선택: 상세 기준, 문서, 정책
        └── assets/           # 선택: 템플릿, 샘플, 리소스
2

용도

Skills는 “자주 반복하는 프롬프트”를 “재사용 가능한 업무 절차”로 바꾸는 데 적합하다.

  • 반복 지시 제거 — 매번 붙여 넣던 기준, 체크리스트, 출력 형식을 Skill로 분리한다.
  • 결과물 표준화 — 보고서, 리뷰 코멘트, 테스트 케이스처럼 산출물 형식을 일정하게 유지한다.
  • 팀 규칙 재사용 — 프로젝트 규칙, 코딩 컨벤션, 보안 기준, 운영 절차를 작업 단위로 묶는다.
  • 스크립트와 결합 — 단순 지침을 넘어 빌드, 테스트, 포맷팅, 로그 수집 같은 명령어를 함께 사용할 수 있게 한다.
판단 기준 특정 지시를 세 번 이상 반복해서 붙여 넣고 있다면 Skill 후보로 볼 수 있다. 단, 너무 넓은 Skill 하나로 모든 일을 처리하려고 하면 오히려 재사용성이 떨어진다.
3

종류

Skills는 도구 종류라기보다 “반복되는 작업 방식”을 어떻게 묶느냐에 따라 나눌 수 있다.

유형 사용 상황 포함 요소
Instruction Skill 명령어 없이 작업 기준과 절차만 반복할 때 작업 순서, 판단 기준, 출력 형식
Reference Skill 긴 규칙·정책·문서 기준을 필요할 때만 불러와야 할 때 references 폴더, 정책 문서, 예시 자료
Script Skill 빌드, 테스트, 변환, 분석처럼 반복 실행할 명령이 있을 때 scripts 폴더, 실행 명령, 입력/출력 규칙
Template Skill 보고서, 리뷰, 릴리즈 노트처럼 일정한 산출물을 만들 때 문서 템플릿, 샘플 출력, 작성 규칙
4

핵심 실무 패턴

Skills는 개수를 많이 나열하기보다, 반복 가치가 높은 작업 몇 개를 정확히 설계하는 편이 낫다.

아래 네 가지는 실무에서 Skill로 만들었을 때 재사용 가치가 높은 패턴이다. 공통적으로 반복 상황, Skill에 넣을 기준, 예시 산출물 을 함께 정의해야 한다.

Pattern 01

요구사항·문서 분석 Skill

  • 반복 상황 RFP, 기획서, 회의록, 화면 정의서를 자주 검토할 때.
  • 넣을 기준 기능/비기능 요구사항 분리, 모호한 표현 탐지, 확인 질문 생성 규칙.
  • 결과물 요구사항 목록, 누락 항목, 확인 질문, 위험도 표.
Pattern 02

코드 리뷰·품질 점검 Skill

  • 반복 상황 PR 리뷰, 리팩터링 검토, 사내 코딩 규칙 확인이 반복될 때.
  • 넣을 기준 보안, 예외 처리, 성능, 가독성, 테스트 누락, 프로젝트 컨벤션.
  • 결과물 파일별 리뷰 의견, 위험도, 수정 우선순위, 개선 코드 예시.
Pattern 03

테스트 생성·검증 Skill

  • 반복 상황 기능 구현 후 테스트 케이스와 검증 절차를 매번 작성할 때.
  • 넣을 기준 정상/예외/경계값, 권한 조건, 실패 조건, 테스트 명명 규칙.

3.3

Plugin 이해하기

Plugin은 MCP처럼 외부 기능을 연결할 수도 있지만, 핵심은 “특정 제품이나 실행 환경에 기능을 추가하는 확장 단위”라는 점이다.

1 개념

Plugin 은 특정 도구나 제품 안에서 사용할 수 있는 기능 확장 패키지다. AI Agent 관점에서는 slash command, hook, sub-agent, MCP server, skill, workflow 등을 하나의 설치 단위로 묶어 제공하는 방식으로 볼 수 있다.

MCP가 “외부 시스템과 연결하는 표준 인터페이스”에 가깝고, Skills가 “반복 작업 방식을 패키징하는 구조”에 가깝다면, Plugin은 이 기능들을 특정 실행 환경 안에서 설치하고 호출할 수 있게 묶는 배포·확장 단위 에 가깝다.

핵심 Plugin은 단일 기능이 아니라 여러 확장 요소를 묶을 수 있다. 예를 들어 Claude Code plugin은 commands, agents, hooks, MCP servers 등을 포함할 수 있고, Cursor plugin도 rules, MCP servers, commands, skills 같은 요소를 marketplace 형태로 제공할 수 있다.

2 용도

Plugin은 반복적으로 필요한 확장 기능을 프로젝트나 팀 단위로 공유할 때 유용하다. 매번 MCP 서버를 따로 연결하거나 명령어를 직접 작성하는 대신, 필요한 기능을 하나의 plugin으로 설치해 같은 환경에서 재사용할 수 있다.

  • 팀 표준화 — 팀에서 쓰는 commands, hooks, agents, MCP 서버를 같은 방식으로 배포한다.
  • 개발 환경 확장 — IDE, CLI, 에디터 안에 특정 업무용 기능을 추가한다.
  • 도구 묶음 배포 — GitHub, Slack, Figma, DB, 브라우저 등 여러 연결 요소를 하나의 설치 단위로 묶는다.
  • 업무 흐름 자동화 — 코드 리뷰, PR 생성, 배포 점검, 문서화처럼 반복되는 흐름을 명령어나 hook으로 만든다.

4 Claude 기준 실무 활용 유형

Claude Code에서 plugin은 command, agent, hook, MCP server, skill을 각각 따로 설치하는 것이 아니라, 팀에서 반복 사용하는 개발 워크플로를 하나의 설치 가능한 패키지로 묶는 방식으로 이해하는 것이 좋다.

핵심 관점 Plugin은 단일 기능이 아니라 팀 단위 배포 패키지 다. commands, hooks, agents, MCP servers, skills를 함께 묶어 프로젝트마다 같은 작업 흐름을 재사용하게 만든다.

팀 표준 명령 패키지

PR 요약, 릴리즈 노트 작성, 테스트 계획 생성, 코드 리뷰 요청처럼 팀에서 반복 사용하는 slash command를 plugin으로 묶는다. 개인별 프롬프트에 의존하지 않고, 같은 명령 이름과 같은 출력 형식을 재사용하는 데 목적이 있다.

검증·보안 Hook 패키지

민감 파일 수정 전 경고, secret 노출 감지, 테스트 미실행 상태에서 완료 방지, 특정 명령 실행 전 확인처럼 작업 전후 검증 규칙을 자동화한다. Plugin으로 묶으면 프로젝트마다 동일한 안전장치를 배포할 수 있다.

도구 연결 패키지

GitHub MCP, Figma MCP, filesystem MCP 같은 도구 연결과 관련 command, skill, hook을 함께 묶는다. 예를 들어 GitHub 작업용 plugin은 GitHub MCP, PR 요약 command, 리뷰 기준 skill, merge 전 hook을 하나의 패키지로 구성할 수 있다.

역할 기반 Agent 패키지

security-reviewer, frontend-reviewer, test-writer, migration-checker처럼 특정 역할을 수행하는 subagent를 plugin으로 배포한다. 팀은 프로젝트 성격에 맞는 agent 세트를 설치해 리뷰, 테스트, 마이그레이션 점검을 표준화할 수 있다.

3.4

차이와 선택 기준

MCP, Skills, Plugin은 모두 AI Agent를 확장하는 방법이지만, 확장하는 대상이 다르다. 핵심은 “무엇을 확장하려는가”를 먼저 구분하는 것이다.

1 한눈에 보는 차이

세 개념은 서로 대체 관계라기보다 역할이 다른 확장 방식이다. MCP는 외부 시스템 접근을 열어주고, Skills는 반복 작업 방식을 표준화하며, Plugin은 특정 제품이나 실행 환경에 확장 기능을 설치·배포하는 단위에 가깝다.

구분 MCP Skills Plugin
핵심 역할 외부 도구·데이터 연결 반복 작업 방식 패키징 제품·환경 기능 확장
중심 질문 AI가 무엇에 접근해야 하는가? AI가 어떤 기준과 절차를 반복해야 하는가? 어떤 환경에 기능을 설치·배포할 것인가?
주요 예시 GitHub, Figma, DB, 브라우저, 파일시스템 요구사항 분석, 코드 리뷰, 테스트 생성, 배포 점검 Claude Code plugin, Cursor plugin, GPT Actions
주의점 권한, 인증, 데이터 접근 범위 프로젝트별 규칙 차이, 과도한 범용화 제품 종속성, 설치 출처, 실행 스크립트

2 언제 무엇을 쓰는가

  • 외부 시스템에 접근해야 한다면 MCP — GitHub 이슈, Figma 디자인, DB, 브라우저, 사내 API처럼 Agent가 직접 읽거나 호출해야 하는 대상이 있을 때 적합하다.
  • 반복 업무 기준을 재사용하려면 Skills — 코드 리뷰 기준, 테스트 작성 절차, 요구사항 분석 방식처럼 프로젝트별 작업 기준을 반복 적용해야 할 때 적합하다.
  • 팀 단위로 기능을 설치·배포하려면 Plugin — commands, hooks, agents, MCP servers, skills를 하나의 패키지로 묶어 같은 개발 환경에 배포할 때 적합하다.
판단 기준 선택 기준은 “무엇을 확장하느냐”다. 외부 데이터·도구 연결은 MCP, 반복 작업 방식의 재사용은 Skills, 특정 실행 환경에 붙는 기능 확장은 Plugin으로 보면 된다.

3 조합 예시

실무에서는 셋 중 하나만 쓰기보다 함께 쓰는 경우가 많다. 예를 들어 GitHub 이슈 기반으로 코드를 수정하고 PR을 점검하는 흐름을 만든다면, GitHub 정보 접근은 MCP가 담당하고, 리뷰 기준과 테스트 생성 기준은 Skills로 정의하며, 이 전체 흐름을 팀 공통 명령과 hook으로 묶으면 Plugin이 된다.

문서 검색 MCP로 최신 자료를 찾고, 요약·분류 Skill로 산출물 형식을 고정한 뒤, Plugin으로 반복 명령을 제공한다.

4 보안과 운영 주의사항

Extension은 Agent의 능력을 넓히지만, 동시에 접근 권한과 실행 범위도 넓힌다. 특히 MCP 서버와 Plugin은 파일, API, DB, 브라우저, SaaS 계정 권한을 다룰 수 있으므로 설치 전 출처와 권한 범위를 확인해야 한다.

  • 최소 권한 — 처음부터 쓰기 권한을 주지 말고, 가능하면 읽기 전용 범위부터 연결한다.
  • 출처 확인 — 공식 문서, GitHub 업데이트 이력, maintainer, issue 상태를 확인한다.
  • 토큰 관리 — API token, PAT, secret이 코드나 로그에 노출되지 않도록 관리한다.
  • 팀 표준화 — 개인별 임의 설정보다 검토된 MCP 서버, Skills, Plugin을 공통 템플릿으로 관리한다.
정리 MCP는 연결 , Skills는 작업 방식 , Plugin은 배포 가능한 확장 패키지 다. 세 개념을 구분하면 Agent를 무작정 복잡하게 만드는 대신, 필요한 지점에 필요한 확장만 붙일 수 있다.

4.1

Harness Engineering

Harness Engineering은 AI Agent를 모델 하나로 보지 않고, 모델을 둘러싼 실행 루프, 도구 호출, 컨텍스트 관리, 상태 관리, 제어 흐름, 기록 방식을 함께 설계하는 접근이다.

1 정의 앵커

Harness Engineering을 이해할 때 가장 먼저 잡아야 할 기준은 AI Agent = Model + Harness 라는 관점이다. 여기서 Model은 Claude, GPT, Gemini 같은 LLM이고, Harness는 모델을 제외한 코드, 설정, 실행 루프, 도구 호출, 컨텍스트 관리, 상태 관리, 제어 흐름, 기록 로직 전체를 의미한다.

Core Equation AI Agent = Model + Harness
모델이 답을 생성한다면, Harness는 그 모델이 어떤 정보를 보고, 어떤 순서로 일하고, 어떤 도구를 쓰고, 언제 멈추고, 무엇을 기록할지 정하는 실행 구조다.

따라서 Harness는 단순 보조 장치가 아니다. AI Agent가 실제 업무를 수행할 수 있도록 모델의 출력 능력을 실행 가능한 작업 능력으로 바꾸는 바깥 구조다.

2 왜 필요한가

같은 모델을 사용하더라도 Harness가 없으면 결과가 매번 흔들릴 수 있다. AI Agent가 어떤 컨텍스트를 기준으로 판단했는지, 어떤 도구를 언제 호출했는지, 현재 작업 상태가 어디인지, 결과가 완료 조건을 만족했는지 추적할 수 없기 때문이다.

  • 실행 루프가 없는 문제 — 관찰, 계획, 실행, 검증 흐름이 없으면 AI Agent가 바로 행동하고 결과만 보고할 수 있다.
  • 도구 호출이 불안정한 문제 — 어떤 상황에서 어떤 도구를 사용할지 정해져 있지 않으면 불필요하거나 위험한 호출이 발생할 수 있다.
  • 컨텍스트 선택이 흔들리는 문제 — 필요한 문서와 파일을 단계별로 선택하지 않으면 오래된 정보나 무관한 정보에 의해 결과가 왜곡될 수 있다.
  • 상태와 완료 판단이 약한 문제 — 현재 진행 단계, 실패 이력, 남은 작업, 완료 기준이 없으면 그럴듯한 보고로 끝날 수 있다.
동기 Harness 설계는 모델 선택만큼이나 AI Agent의 결과 품질과 안정성에 큰 영향을 준다. 특히 복잡한 작업에서는 모델 자체보다 실행 루프, 컨텍스트 관리, 도구 호출 기준, 상태 추적 방식이 결과를 좌우하는 경우가 많다.

3 Harness 핵심 구성요소

Harness의 핵심은 거버넌스나 보안 기준만이 아니다. AI Agent가 반복적으로 작업을 수행할 수 있도록 루프, 도구, 컨텍스트, 상태, 제어 흐름, 기록 이 하나로 연결되어야 한다.

구성요소 역할 설계 질문
실행 루프 AI Agent가 관찰하고, 계획하고, 실행하고, 검증하는 반복 흐름을 만든다. 작업은 어떤 단계로 반복되는가?
도구 호출 파일, CLI, API, 브라우저, 테스트 명령 등을 언제 사용할지 정한다. 어떤 조건에서 어떤 도구를 호출하는가?
컨텍스트 관리 작업 단계마다 필요한 문서, 코드, 로그, 요구사항을 선별한다. AI Agent가 지금 봐야 할 정보는 무엇인가?
상태·메모리 현재 단계, 결정 사항, 실패 이력, 남은 작업을 추적한다. 현재 작업 상태를 어떻게 유지하는가?
제어 흐름 다음 단계로 진행할지, 재시도할지, 중단할지 결정하는 기준을 둔다. 언제 다음 단계로 넘어가고 언제 멈추는가?
경계·권한 수정 가능한 범위, 실행 가능한 명령, 승인 필요한 작업을 제한한다. AI Agent가 해도 되는 일과 안 되는 일은 무엇인가?
실패 처리 실패 원인을 분류하고 재시도, 중단, 사람 개입 기준을 적용한다. 실패했을 때 어떻게 대응하는가?
기록·추적 실행 명령, 변경 파일, 판단 근거, 검증 결과를 남긴다. 나중에 무엇을 추적할 수 있어야 하는가?

4 Harness 예시

Harness 예시는 특정 도구 하나가 아니라, AI Agent가 작업을 수행하는 전체 실행 흐름으로 보는 것이 좋다. 아래 예시들은 실행 루프, 도구 호출, 컨텍스트 관리, 상태 관리가 어떻게 묶이는지 보여준다.

Example 01

코드 수정 Harness

버그 수정이나 기능 변경을 AI Agent에게 맡길 때, 작업 범위와 검증 기준을 함께 설계하는 예시다. 

1. 수정 요청과 관련 요구사항 확인
2. 관련 파일 후보와 영향 범위 정리
3. 수정 계획 작성
4. 필요한 파일만 수정
5. 테스트 또는 빌드 실행
6. 실패 시 원인 분류 후 재시도 또는 중단
7. 변경 요약과 검증 결과 기록
이 예시에서 요구사항 확인은 컨텍스트 관리, 파일 후보 정리는 상태 관리, 테스트 실행은 검증 루프, 실패 분류는 제어 흐름에 해당한다.
Example 02

문서 기반 구현 Harness

SDD, 요구사항 문서, 회의록 같은 문서를 기준으로 기능을 구현할 때 사용하는 예시다. 

1. 문서에서 요구사항, 제약, 완료 기준 추출
2. 구현 범위를 작업 단위로 분리
3. 각 작업의 입력 문서와 관련 파일 연결
4. 구현 후 요구사항과 결과 비교
5. 누락, 모호함, 문서-코드 불일치 기록
6. 사람 확인이 필요한 항목 분리
문서 기반 구현 Harness는 “문서를 읽고 구현하기”가 아니라, 문서의 요구사항이 코드 결과와 어떻게 연결되는지 추적하는 구조다.

5 Harness를 설계할 때 챙겨야 하는 것

Harness를 설계할 때는 기능 목록보다 실행 기준을 먼저 정해야 한다. 아래 항목이 정리되어 있어야 AI Agent의 작업을 통제하고 개선할 수 있다.

항목 질문 정리해야 할 내용
작업 범위 AI Agent가 무엇을 해야 하는가? 대상 기능, 제외 범위, 수정 가능한 파일 범위
완료 기준 언제 작업이 끝났다고 볼 것인가? 테스트 통과, 문서 충족, 리뷰 완료, 산출물 형식
실행 루프 어떤 순서로 작업을 반복할 것인가? Observe, Plan, Act, Verify 단계와 반복 조건
도구 사용 어떤 도구를 언제 사용할 것인가? 도구 호출 조건, 허용 명령, 실패 시 대체 경로

기록 방식 무엇을 남길 것인가? 실행 명령, 변경 파일, 판단 근거, 검증 결과

6 Harness Engineering이 아닌 것

주의 아래 요소들은 Harness의 재료가 될 수는 있지만, 그 자체만으로 Harness Engineering이라고 보기는 어렵다. 핵심은 이 요소들이 하나의 실행 루프와 제어 흐름으로 연결되어 있는가다.
  • 프롬프트를 길게 쓰는 것만 — 지시문은 재료일 뿐, 실행 상태와 완료 기준을 대신하지 못한다.
  • 좋은 모델을 고르는 것만 — 모델 선택은 중요하지만, 도구 호출과 실패 처리 구조를 대체하지 못한다.
  • 도구를 많이 연결하는 것만 — 도구가 많아도 호출 조건과 권한 범위가 없으면 통제하기 어렵다.
  • Plugin이나 자동화를 설치하는 것만 — 설치된 기능이 작업 루프와 상태 관리로 연결되지 않으면 실행 구조가 되지 않는다.
  • 체크리스트를 문서로 적어두는 것만 — 문서화된 기준이 실제 실행·검증·기록 흐름에 반영되어야 한다.

7 좋은 Harness와 나쁜 Harness

앞 항목이 “무엇을 Harness로 착각하면 안 되는가”를 다뤘다면, 여기서는 실제 Harness의 설계 품질을 비교한다. 나쁜 Harness는 재료가 부족한 것이 아니라, 재료들이 실행 루프로 연결되지 않은 상태에 가깝다.

구분 나쁜 Harness 좋은 Harness
작업 루프 목표를 받은 뒤 바로 실행하고 결과만 보고한다. 관찰, 계획, 실행, 검증 흐름을 분리한다.
컨텍스트 선택 어떤 파일과 문서를 봐야 하는지 기준이 없다. 작업 단계마다 필요한 컨텍스트를 선택하고 갱신한다.
도구 호출 도구를 언제 호출할지, 실패하면 어떻게 할지 기준이 없다. 도구 호출 조건, 권한 범위, 대체 경로를 둔다.
상태 추적 현재 단계, 결정 사항, 남은 작업이 추적되지 않는다. 진행 상태와 중간 결정을 기록하고 다음 단계 판단에 사용한다.
실패 대응 실패 원인 분류 없이 계속 시도하거나 임의로 포기한다. 재시도, 중단, 사람 개입 기준을 둔다.
기록 최종 결과만 남긴다. 판단 근거, 실행 명령, 변경 파일, 검증 결과를 남긴다.
4.2

Agent Orchestration

Agent Orchestration은 AI Agent가 복잡한 작업을 한 번에 처리하지 않도록 실행 순서, 도구 호출, 컨텍스트 범위, 위임 구조, 검증 조건을 조율하는 방식이다.

1 왜 Orchestration이 필요한가

복잡한 작업에서 문제는 답을 생성하는 것만이 아니다. AI Agent가 무엇을 먼저 확인하고, 어떤 기준으로 계획하고, 어느 도구를 호출하고, 언제 검증하거나 멈출지를 정하지 않으면 실행이 불안정해진다.

핵심 Orchestration은 AI Agent의 작업을 관찰, 계획, 실행, 검증 으로 나누고, 각 단계의 책임과 다음 단계로 넘어갈 조건을 정하는 설계다.

2 Orchestration이 조율하는 대상

Orchestration은 단순히 순서를 나열하는 것이 아니라, AI Agent가 작업 중 어떤 판단을 어디서 해야 하는지를 정한다.

조율 대상 설명 설계 질문
작업 순서 관찰, 계획, 실행, 검증 단계를 어떤 흐름으로 진행할지 정한다. 무엇을 먼저 확인해야 하는가?
도구 호출 파일 읽기, 코드 수정, 테스트 실행, 검색, API 호출을 어느 단계에서 사용할지 정한다. 어떤 조건에서 도구를 호출할 것인가?
컨텍스트 범위 각 단계에서 필요한 문서, 코드, 로그, 요구사항만 선별해 전달한다. 지금 단계에서 꼭 필요한 정보는 무엇인가?
상태 전환 계획 중, 실행 중, 검증 중, 완료, 실패 같은 상태를 관리한다. 언제 다음 단계로 넘어갈 수 있는가?
위임 여부 메인 AI Agent가 직접 처리할지, Sub Agent에게 하위 작업을 넘길지 결정한다. 나눠서 처리할 만큼 독립적인 작업인가?
검증 기준 결과가 완료 조건을 만족하는지 확인하고 재시도, 중단, 완료를 결정한다. 성공과 실패를 무엇으로 판단할 것인가?

3 Sub Agent 개념

Sub Agent는 메인 AI Agent가 모든 작업을 하나의 컨텍스트에서 처리하지 않고, 특정 역할이나 하위 작업을 별도 지시와 별도 컨텍스트로 분리해 처리하도록 위임하는 단위다.

Role Split

역할 분리

분석, 구현, 리뷰, 테스트처럼 작업 성격을 나누어 각각의 책임을 명확히 한다.

Context Isolation

컨텍스트 격리

모든 정보를 한 번에 몰아넣지 않고, 하위 작업에 필요한 정보만 전달한다.

Delegation

작업 위임

메인 AI Agent가 직접 처리하기보다 독립적으로 조사하거나 검토할 수 있는 일을 맡긴다.

Aggregation

결과 취합

Sub Agent의 결과를 메인 AI Agent가 받아 계획, 실행, 검증 흐름에 다시 반영한다.

구분 Sub Agent는 보통 하나의 상위 작업 안에서 메인 AI Agent가 하위 역할을 위임하는 구조다. Multi-Agent Orchestration은 여러 AI Agent 또는 여러 모델 기반 Agent가 더 독립적인 역할과 상호작용 구조를 가지고 협업하는 방식이다.

4 Observe → Plan → Act → Verify 실행 루프

단일 AI Agent 흐름에서도 Orchestration은 필요하다. 핵심은 단계를 나누고, 각 단계에서 어떤 판단을 할지 정하는 것이다.

Observe

관찰

요구사항, 파일, 오류, 로그, 제약을 확인한다.

→
Plan

계획

직접 처리할 작업과 위임할 작업을 나눈다.

→
Act

실행

코드 수정, 문서 작성, 도구 호출, Sub Agent 위임을 수행한다.

→
Verify

검증

완료 기준을 확인하고 재시도, 중단, 완료를 결정한다.

5 Sub Agent 위임 구조 예시

위임 구조를 설계할 때는 “누가 무엇을 맡고, 어떤 결과를 누구에게 돌려주는가”가 명확해야 한다. 아래는 하나의 메인 AI Agent가 하위 분석 작업을 Sub Agent에게 나누어 맡기는 구조다.

Main AI Agent

목표·완료 기준 관리

전체 요청을 해석하고, 위임할 작업을 나누고, 최종 결과를 판단한다.

Sub Agent A

파일·영향 범위 조사

관련 파일, 의존성, 변경 영향 범위를 정리한다.

Sub Agent B

테스트 실패 분석

실패 로그와 재현 조건을 확인하고 원인을 분류한다.

Sub Agent C

리뷰 포인트 정리

변경 위험, 누락 가능성, 확인해야 할 기준을 정리한다.

↓ 결과 취합
Main AI Agent

수정 계획 또는 최종 답변 생성

Sub Agent 결과를 비교하고 충돌을 정리한 뒤 다음 실행 또는 완료 보고를 결정한다.

Example

코드 변경 영향 분석 위임

큰 코드베이스에서 메인 AI Agent가 모든 파일과 테스트를 한 번에 보지 않고, 하위 분석을 나누어 처리하는 예시다. 

1. Main AI Agent: 변경 목표와 완료 기준 정의
2. Sub Agent A: 관련 파일과 영향 범위 조사
3. Sub Agent B: 테스트 실패 가능성과 검증 기준 조사
4. Sub Agent C: 리뷰 포인트와 위험 구간 정리
5. Main AI Agent: 결과 취합 후 수정 계획 작성
6. Main AI Agent: 실행 또는 사람 승인 요청 결정
이 구조의 핵심은 병렬로 많이 시키는 것이 아니라, 하위 작업의 입력·출력·판단 기준을 분명히 정하는 것이다.

6 설계 시 주의할 점

  • 작은 작업은 나누지 않기 — 단순 수정이나 짧은 문서 작업까지 Sub Agent로 나누면 비용과 복잡도만 늘어난다.
  • 위임 입력 명확히 하기 — Sub Agent에게 무엇을 보고, 무엇을 만들고, 어떤 형식으로 반환할지 정해야 한다.
  • 최종 판단자 정하기 — Sub Agent 결과가 다를 때 누가 취합하고 결정할지 정해야 한다.
  • 컨텍스트 중복 줄이기 — 모든 Sub Agent에게 같은 전체 정보를 넘기면 컨텍스트 격리의 장점이 사라진다.
  • 결과 취합 기준 두기 — 하위 결과를 그대로 합치지 말고 충돌, 누락, 우선순위를 정리해야 한다.
4.3

Multi-Agent Orchestration

Multi-Agent Orchestration은 여러 AI Agent가 역할을 나누어 하나의 목표를 수행하도록 작업 분배, 실행 순서, 결과 전달, 취합 방식을 조율하는 구조다.

1 개념과 범위

Multi-Agent Orchestration 자체는 공식 프레임워크와 아키텍처 문서에서도 다루는 개념이다. Supervisor, Handoff, Concurrent, Group Chat처럼 여러 AI Agent의 역할과 흐름을 조율하는 패턴이 여기에 포함된다.

다만 Claude Code, Codex CLI, Gemini CLI처럼 서로 다른 제품이나 CLI를 사용자가 엮어 협업시키는 방식은 별도로 구분해야 한다. 이 방식은 Multi-Agent Orchestration의 응용 예시로 볼 수 있지만, 특정 벤더가 공통 표준으로 보장하는 공식 방식이라기보다는 Cross-CLI / Cross-Model 협업 패턴 에 가깝다.

핵심 구분 Multi-Agent Orchestration 은 공식적으로 다뤄지는 설계 개념이고, Claude · Codex · Gemini를 엮는 방식 은 사용자가 여러 도구를 조합해 만드는 Cross-CLI / Cross-Model 응용 패턴으로 보는 것이 안전하다.

2 공식적으로 다뤄지는 Orchestration 패턴

공식 문서와 프레임워크에서는 여러 AI Agent를 조율하는 방식을 몇 가지 패턴으로 설명한다. 중요한 것은 Agent의 개수가 아니라, 누가 작업을 나누고 누가 결과를 취합하며 누가 최종 응답을 소유하는가다.

패턴 구조 적합한 상황
Supervisor 상위 Orchestrator가 여러 전문 AI Agent에게 작업을 나누고 결과를 취합한다. 역할 분담과 최종 판단자가 명확해야 할 때
Handoff 현재 AI Agent가 다음 AI Agent에게 대화나 작업 소유권을 넘긴다. 단계별 전문성이 다르고 담당자가 바뀌어야 할 때
Concurrent 여러 AI Agent가 병렬로 같은 문제의 다른 부분을 처리한다. 분석, 검토, 테스트처럼 독립 처리가 가능한 작업
Group Chat 여러 AI Agent가 같은 대화 공간에서 의견을 주고받고 결론을 만든다. 토론, 비교, 의사결정 후보를 만들 때

3 Cross-CLI / Cross-Model 협업 패턴

Claude Code, Codex CLI, Gemini CLI처럼 서로 다른 AI CLI나 모델을 함께 쓰는 방식은 공식 Multi-Agent 패턴을 사용자가 실무 도구 조합으로 응용한 형태에 가깝다. 이때 핵심은 “여러 개를 동시에 켠다”가 아니라, 역할과 산출물 전달 규칙을 정하는 것이다.

Cross-CLI / Cross-Model View

4 공식 패턴과 사용자 조합 방식의 차이

구분 공식 Multi-Agent 패턴 Cross-CLI / Cross-Model 조합
관리 주체 프레임워크나 제품이 Agent 호출, handoff, 상태 일부를 관리한다. 사용자, 스크립트, 별도 control plane이 전달과 취합을 관리한다.
상태 관리 공유 상태, memory, run trace를 프레임워크가 제공할 수 있다. 파일, 문서, handoff note, git history로 상태를 직접 남겨야 한다.
전달 방식 handoff, tool call, supervisor 호출 같은 내장 패턴을 쓴다. CLI 출력, markdown 요약, commit diff, issue comment 등을 전달 형식으로 쓴다.
주의점 패턴을 잘못 고르면 Agent가 불필요하게 많아지고 비용이 늘어난다. 파일 충돌, 중복 작업, 서로 다른 판단 기준, 최종 책임 부재가 생기기 쉽다.

5 설계 시 주의할 점

State

상태를 남기기

각 AI Agent가 무엇을 했고 무엇이 남았는지 handoff note나 작업 로그로 남긴다.

Conflict

파일 충돌 관리

여러 AI Agent가 같은 파일을 수정할 수 있으므로 branch, worktree, diff 확인 기준을 둔다.

Owner

최종 판단자 지정

여러 결과가 나와도 최종 승인자는 사람 또는 명시된 Reviewer가 맡아야 한다.

Boundary

역할 경계 정하기

Claude, Codex, Gemini의 역할이 겹치면 중복 작업과 판단 충돌이 늘어난다.
